# Система событий

## Обзор

Система случайных событий добавляет непредсказуемость и реиграбельность в игру. События происходят автоматически и могут влиять на различные аспекты жизни персонажа.

## Моменты срабатывания

События могут срабатывать в следующих моментах:
- После завершения действия игрока (базовые шансы зависят от типа действия)
- В конце игровой недели (обязательно 1 событие)
- При достижении определённого возраста или количества игровых дней

## Категории событий

### По эффекту:
- **Положительные** — дают бонусы (например, +30% к ЗП за период, +15 Настроение)
- **Отрицательные** — налагают штрафы (например, −10 Здоровья, −5 000 ₽ недополученной ЗП)
- **Нейтральные / сюжетные** — предлагают выбор из 2–3 вариантов с разными последствиями
- **Возрастные** — появляются после 30, 40, 50, 60 лет (связаны со здоровьем или семьёй)

### По вероятности возникновения:
- Частые события
- Средние события  
- Редкие события

## Механика микро-событий

Микро-события могут происходить после каждого пользовательского действия (кроме сна).

Формула расчёта шанса события:
```
finalChance = baseChance(action,event) * riskModifiers * skillModifiers * stateModifiers
```

Где:
- `baseChance(action,event)` — базовый шанс события для конкретного действия и события
- `riskModifiers` — риск-факторы (дорогие вещи на персонаже, опасный район, позднее время)
- `skillModifiers` — навыки и параметры (например, физическая форма, атлетизм)
- `stateModifiers` — текущее состояние (усталость, стресс, здоровье и т.д.)

Важно разделять:
- шанс того, что событие вообще произойдёт;
- шансы исходов события (благополучно, потери, травма, побег и т.д.).

## Структура событий

События хранятся в JSON-формате с полями:
- `id` — уникальный идентификатор события
- `title` — заголовок события
- `description` — описание события
- `triggerCondition` — условие срабатывания
- `choices[]` — массив вариантов выбора

Каждый `choice` содержит:
- Текст выбора
- `effect` — изменение параметров (деньги, шкалы, навыки и т.д.)

## Техническая реализация

### ECS Компоненты

События представлены в ECS через два основных компонента:

1. **eventQueue** — очередь ожидающих событий
   ```typescript
   eventQueue: {
     pendingEvents: EventQueueItem[]; // события, ожидающие показа
     seenInstanceIds: Set<string>;   // уже показанные события (для дедупликации)
     processedWeeklyEvents: Set<string>;  // обработанные недельные события
     processedMonthlyEvents: Set<string>; // обработанные месячные события
     processedYearlyEvents: Set<string>;  // обработанные годовые события
   }
   ```

2. **eventHistory** — история уже发生的 событий
   ```typescript
   eventHistory: {
     events: EventHistoryEntry[]; // список уже произошедших событий
   }
   ```

### Форматы событий

**Legacy формат (v1)** - использовался в ранних версиях:
```typescript
// EventQueueItem
{
  id: string
  type: string
  title: string
  description: string
  choices?: EventChoice[]
  data?: Record<string, unknown>
  day: number
}

// EventHistoryEntry
{
  eventId: string
  day: number
  choiceId?: string
  effects?: Record<string, number>
}
```

**Canonical формат (v2)** - текущий формат с улучшенной дедупликацией:
```typescript
// EventQueueItem
{
  id: string
  instanceId: string  // Уникальный идентификатор экземпляра события
  type: string
  title: string
  description: string
  choices?: EventChoice[]
  data?: Record<string, unknown>
  day: number
  week?: number     // Номер недели
  month?: number    // Номер месяца
  year?: number     // Номер года
  _priority?: string  // Приоритет обработки
  _source?: string  // Источник события
}

// EventHistoryEntry
{
  instanceId: string  // Основной ключ (был eventId в v1)
  templateId: string  // ID шаблона события (был eventId в v1)
  day: number
  week?: number
  month?: number
  year?: number
  choiceId?: string
  choiceText?: string  // Текст выбранного варианта
  effects?: Record<string, number>
  resolvedAt?: number  // Время разрешения события (timestamp)
}
```

### Процесс миграции

При загрузке сохранения происходит автоматическая миграция с legacy формата v1 в canonical формат v2:

1. Проверка версии сохранения
2. Если версия < 1.2.0, применяется миграция событий
3. Legacy события преобразуются в canonical формат
4. Генерация `instanceId` для legacy событий по формуле:
   ```
   instanceId = {eventId}_{totalHours}_{timestamp}
   ```
5. Для новых событий используется детерминированный генератор:
   ```
   instanceId = {templateId}_{totalHours}_{sequence}
   ```
6. Инициализация новых структур: `seenInstanceIds`, `processedWeeklyEvents`, `processedMonthlyEvents`, `processedYearlyEvents`
7. Обновление `eventSchemaVersion` до 2

### ECS Системы

**EventQueueSystem** отвечает за:
- Добавление новых событий в очередь
- Определение времени срабатывания событий
- Фильтрацию уже показанных событий (дедупликация)
- Обновление таймеров событий

**EventChoiceSystem** обрабатывает:
- Отображение интерфейса выбора события
- Обработку выбора пользователем варианта события
- Применение эффектов выбранного варианта

**EventHistorySystem** управляет:
- Записью произошедших событий в историю
- Предотвращением слишком częстых повторений одних и тех же событий
- Хранением информации о выбранных вариантах и применённых эффектах

## Взаимодействие с другими системами

- **TimeSystem**: Определяет моменты срабатывания событий (конец недели, месяца, года)
- **ActionSystem**: Триггерит проверку на микро-события после действий
- **SkillsSystem**: Навыки модифицируют шансы и исходы событий
- **StatsSystem**: Шкалы персонажа влияют на состояние, которое модифицирует шансы событий
- **CareerProgressSystem**: Некоторые события зависят от карьерного прогресса
- **FinanceActionSystem**: Финансовые события влияют на кошелёк и дают бонусы к финансовой грамотности

## Визуализация

События отображаются в модальном окне с:
- Текстом события
- Описанием ситуации
- Кнопками выбора вариантов действий
- После выбора показываются результаты выбранного варианта

## Баланс событий

Текущий баланс по категориям:
- Положительные: 7 событий (28%)
- Отрицательные: 6 событий (24%)
- Нейтральные: 4 события (16%)
- Возрастные: 8 событий (32%)

В среднем игрок столкнётся с каждым событием 2–3 раза за игру.
Возрастные события срабатывают ровно 1 раз за игру.
Общий пул событий обеспечивает разнообразие на всю игру.
