# План внедрения часовой модели действий

## 1. Цель

Перевести игровой цикл на единую часовую модель, где все действия расходуют время из общего бюджета, а стандартные и крупные события срабатывают на недельных, месячных и годовых границах.

## 2. Зафиксированные правила (Scope Freeze)

1. Базовая единица времени: `1 час`.
2. `1 день = 24 часа`, `1 неделя = 168 часов`.
3. Обязательное время: работа по графику + сон.
4. Сон обязателен, целевое значение: `8 часов/сутки`.
5. Остаток суток не теряется, а закрывается авто-логикой сна/бытового времени.
6. Для безработного пока не вводим отдельные обязательные бытовые часы.
7. Еженедельные события срабатывают в конце недели.
8. Ежемесячные события срабатывают каждые 4 недели.
9. Годовые события срабатывают каждые 12 месяцев.
10. Микро-события могут срабатывать после действий игрока по модели `baseChance + modifiers`.

## 3. Архитектурные инварианты

1. Нельзя выполнить действие, если не хватает часов в текущих сутках.
2. Нельзя выполнить действие, если оно нарушает обязательный рабочий график недели для трудоустроенного персонажа.
3. Все переходы времени идут через единый `TimeService` (без локальных обходов в сценах).
4. Любое завершённое действие обязано писать запись в журнал действий.
5. Любое сработавшее событие обязано писать запись в журнал событий.
6. Все расчёты шансов и исходов событий должны быть детерминированы при одинаковом seed.

## 4. Этапы внедрения

### Этап 0. Подготовка и фиксация контекста

1. Выполнить `npm run mem:mine`, `npm run mem:status`, `npm run mem:wakeup`.
2. Сохранить в MemPalace ключевые решения по часовой модели.
3. Подготовить краткий ADR по формуле времени и событий.
4. Критерий готовности: команда использует один и тот же набор терминов и формул.

### Этап 1. Доменная модель времени

1. Ввести объект времени в состоянии игры: сутки, неделя, месяц, год, часы в текущих сутках и неделе.
2. Ввести поля недосыпа и факт сна за текущие сутки.
3. Ввести API переходов времени: `spendHours`, `endDay`, `endWeek`.
4. Критерий готовности: любое действие меняет время только через единый API.

### Этап 2. Каталог действий и часовая стоимость

1. Перевести все действия на формат `costHours`, `costMoney`, `effects`.
2. Добавить валидацию доступности действия по времени и деньгам.
3. Добавить категории действий: работа, базовые потребности, социальные, обучение, отдых.
4. Критерий готовности: в UI нельзя вызвать действие, не проходящее валидацию.

### Этап 3. Работа и графики

1. Для каждой работы хранить `requiredHoursPerWeek` и `hourlyRate`.
2. Реализовать учёт отработанных часов недели.
3. Реализовать правило недоработки: штраф при незакрытой норме часов на конец недели.
4. Реализовать правило переработки: стресс/усталость выше нормы при избытке часов.
5. Критерий готовности: графики 5/2 и 2/2 корректно ограничивают свободу действий.

### Этап 4. Сон и закрытие дня

1. Ввести действие сна с гибкой длительностью.
2. Ввести штрафы за сон `< 7ч` и усиленные штрафы за сон `<= 6ч`.
3. Ввести автоматическое закрытие суток остатком сна/бытового времени.
4. Критерий готовности: нет состояния, где игрок остаётся в «бесконечном дне».

### Этап 5. Периодические события

1. В конце недели запускать стандартный пакет: зарплата, обязательные расходы, summary.
2. Каждые 4 недели запускать месячный пакет: аренда/коммуналка/большие события.
3. Каждые 12 месяцев запускать годовой пакет: возраст, крупные жизненные события.
4. Критерий готовности: границы периодов не пропускаются и не дублируются.

### Этап 6. Микро-события после действий

1. Реализовать двухшаговую модель: `шанс срабатывания` и `шанс исхода`.
2. Применять формулу: `finalChance = baseChance * riskModifiers * skillModifiers * stateModifiers`.
3. Добавить модификаторы риска от экипировки/контекста действия.
4. Добавить модификаторы исхода от навыков (например, физическая форма и атлетизм).
5. Ввести cooldown на повтор ключевых микро-событий.
6. Критерий готовности: микро-события разнообразны и не абузятся короткими действиями.

### Этап 7. UI/UX изменения

1. В HUD показать: часы суток, часы недели, обязательные часы работы, сон.
2. На карточках действий показать стоимость в часах.
3. Для недоступных действий показывать причину блокировки.
4. Добавить экран недельного и месячного summary.
5. Критерий готовности: игрок всегда понимает, куда ушло время.

### Этап 8. Сохранения и миграция

1. Включить новую структуру времени в save.
2. Включить журнал событий с типом: `micro`, `weekly`, `monthly`, `yearly`, `work`, `story`.
3. Написать мигратор с legacy-формата (`gameDays`, `salaryPerDay`) на часовой формат.
4. Добавить versioning (`1.1.0+`) и безопасный rollback.
5. Критерий готовности: старые сейвы открываются без потери прогресса.

### Этап 9. Баланс и калибровка

1. Прогнать симуляции на 4, 12 и 52 игровых недели.
2. Проверить, что при графике 5/2 свободное время и доход соответствуют ожиданиям.
3. Проверить, что безработный имеет больше времени, но проигрывает по экономике.
4. Откалибровать базовые шансы микро-событий и штрафы недосыпа.
5. Критерий готовности: нет доминирующей «ломающей» стратегии.

### Этап 10. Тестирование и релиз

1. Добавить unit-тесты TimeService, ActionValidator, EventResolver.
2. Добавить интеграционные сценарии на циклы: день, неделя, месяц, год.
3. Добавить regression-набор на загрузку старых сейвов.
4. Подготовить feature flag для безопасного включения.
5. Критерий готовности: зелёный CI и успешный smoke на тестовом профиле.

4. PR-4: Сон, недосып и закрытие суток.
5. PR-5: Еженедельные/месячные/годовые события.
6. PR-6: Микро-события и модификаторы исходов.

8. PR-8: UI/HUD и отчёты.
9. PR-9: Баланс-пасс и тесты.

## 6. Критерии приемки (Definition of Done)

1. Игрок не может бесконечно спамить активности без учёта работы и сна.
2. Любое действие корректно списывает часы и отображается в журнале.
3. Концы дня, недели, месяца и года срабатывают стабильно.
4. Микро-события работают с модификаторами вероятностей и исходов.
5. Старые сейвы мигрируются автоматически.
6. Документация и MemPalace обновлены после внедрения.

## 7. Риски и меры

1. Риск: слишком сложный UX при большом числе временных индикаторов.
2. Мера: показывать ключевые 3 индикатора, остальные в detail-панели.
3. Риск: поломка баланса доход/расход после перехода на ставки в час.
4. Мера: симуляционный прогон и таблица контрольных профилей.
5. Риск: несовместимость legacy-сейвов.
6. Мера: версия формата + мигратор + backup перед конвертацией.

## 8. Операционный workflow с MemPalace

1. Перед каждым этапом: `npm run mem:wakeup`.
2. После завершения этапа: `npm run mem:mine` и `npm run mem:status`.
3. Перед передачей задачи другому ассистенту: прикладывать wakeup + релевантные search-результаты + список затронутых файлов.
