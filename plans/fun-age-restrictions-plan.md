# План: Возрастные ограничения для действий на всех страницах

## Статус: Запланировано (v3 — усилена связность, реализм и архитектурная консистентность)

## Проблема

На всех страницах с карточками действий (`/game/fun`, `/game/health`, `/game/selfdev`, `/game/social`, `/game/hobby`, `/game/shop`, `/game/home`, `/game/career`, `/game/education`, `/game/finance`) отображаются действия для взрослых без ограничений по возрасту. Ни одно из ~200 действий не имеет поля `ageGroup`. В результате **младенец видит и может выполнить** «Шопинг-терапию», «Танцы в клубе», «Открыть бизнес», «Свидание вслепую» и т.д.

Детские действия из `CHILD_ACTIONS` (`child-actions.ts`) имеют `ageGroup`, но они относятся к категориям `fun`, `social`, `basic`, `selfdev`, `health`, `hobby`, `education` только частично.

## Цель

Добавить поле `ageGroup: AgeGroup` ко всем действиям во всех файлах действий, чтобы каждый возраст видел только релевантные действия.

## Важное новое допущение: сон не является action

Сон больше **не рассматривается как обычное игровое действие**. Он должен быть встроен в систему течения времени как **обязательная базовая потребность**, а не как добровольно выбираемая карточка.

Это меняет рамки плана:

1. `fun_sleep_8h` и `fun_short_sleep` не должны участвовать в финальной age-разметке как обычные actions.
2. Базовые потребности делятся на два типа:
   - **автоматические системные**: сон;
   - **явные действия игрока**: прогулка, еда вне дома, уход за собой, бытовые и социальные решения.
3. Критерий «доступно всем, потому что это базовая потребность» больше нельзя автоматически применять к сну.
4. Если в текущем коде sleep-actions пока существуют, их нужно считать **переходным техническим долгом**, а не частью целевой модели.

**Рекомендация:** вынести задачу удаления/замены sleep-actions в отдельный подэтап до финальной чистки ageGroup в `fun-actions.ts`.

## ⚠️ КРИТИЧЕСКИЙ БАГ: Несоответствие маппинга возрастов

### Проблема

В [`useAgeRestrictions/index.ts`](src/composables/useAgeRestrictions/index.ts:98) функция `ageGroup` маппит возраст в `AgeGroup` **без использования `TEEN`**:

```typescript
// Текущий код (БАГОВАННЫЙ):
if (currentAge <= 3)  return AgeGroup.INFANT   // 0
if (currentAge <= 7)  return AgeGroup.TODDLER  // 1
if (currentAge <= 12) return AgeGroup.CHILD    // 2
if (currentAge <= 15) return AgeGroup.KID      // 3 ← 13-15 лет = KID, а не TEEN!
if (currentAge <= 18) return AgeGroup.YOUNG    // 5
return AgeGroup.ADULT                           // 6
```

**`AgeGroup.TEEN (4)` никогда не возвращается!** Это значит:
- Действия с `ageGroup: AgeGroup.TEEN` **никогда не будут показаны** персонажам 13–15 лет
- `isActionAvailable()` проверяет `action.ageGroup <= ageGroup.value` → `TEEN(4) <= KID(3)` = `false`

### Решение (обязательно ДО добавления ageGroup в действия)

Исправить маппинг в **двух местах** одновременно:

**1. [`useAgeRestrictions/index.ts`](src/composables/useAgeRestrictions/index.ts:98) — `ageGroup` computed:**
```typescript
if (currentAge <= 3)  return AgeGroup.INFANT   // 0-3
if (currentAge <= 7)  return AgeGroup.TODDLER  // 4-7
if (currentAge <= 12) return AgeGroup.CHILD    // 8-12
if (currentAge <= 15) return AgeGroup.TEEN     // 13-15 ← ИСПРАВЛЕНО
if (currentAge <= 18) return AgeGroup.YOUNG    // 16-18
return AgeGroup.ADULT                           // 19+
```

**2. [`useAgeRestrictions/index.ts`](src/composables/useAgeRestrictions/index.ts:178) — `getAgeGroup()` helper:**
```typescript
// Та же поправка: ageValue <= 15 → AgeGroup.TEEN
```

**3. [`age-unlocks.ts`](src/composables/useAgeRestrictions/age-unlocks.ts:36) — `getAgeGroup()`:**
```typescript
if (ageValue <= 3)  return 0  // INFANT
if (ageValue <= 7)  return 1  // TODDLER
if (ageValue <= 12) return 2  // CHILD
if (ageValue <= 15) return 4  // TEEN ← ИСПРАВЛЕНО (было 3)
if (ageValue <= 18) return 5  // YOUNG
return 6                       // ADULT
```

**4. [`types.ts`](src/domain/balance/actions/types.ts:7) — исправить комментарии:**
```typescript
export enum AgeGroup {
  INFANT = 0,   // 0-3 года
  TODDLER = 1,  // 4-7 лет  ← было «1-3 года»
  CHILD = 2,    // 8-12 лет ← было «4-7 лет»
  KID = 3,      // НЕ ИСПОЛЬЗУЕТСЯ в маппинге (зарезервировано)

```

**5. [`useAgeRestrictions/index.ts`](src/composables/useAgeRestrictions/index.ts:17) — добавить `AGE_RULES[TEEN]`:**
```typescript

```

## Возрастные группы (исправленная таблица)

| Группа | Возраст | Enum | Примечание |
|--------|---------|------|------------|
| INFANT | 0–3 | `AgeGroup.INFANT = 0` | Младенец, базовые потребности |
| TODDLER | 4–7 | `AgeGroup.TODDLER = 1` | Дошкольник, с родителями |
| CHILD | 8–12 | `AgeGroup.CHILD = 2` | Младший школьник, самостоятельность |
| ~~KID~~ | ~~8–12~~ | `AgeGroup.KID = 3` | **Не используется в маппинге** (историческое значение) |
| TEEN | 13–15 | `AgeGroup.TEEN = 4` | Подросток, осознанные действия |
| YOUNG | 16–18 | `AgeGroup.YOUNG = 5` | Старшеклассник/студент |
| ADULT | 19+ | `AgeGroup.ADULT = 6` | Полная дееспособность |

> В текущей реализации действия **без** `ageGroup` доступны всем возрастам (включая INFANT), но для долгосрочной связности это лучше считать временной совместимостью, а не целевой моделью.

### Принципы назначения ageGroup

1. **Реализм** — основываемся на реальных возрастных ограничениях (закон, физические возможности, социальные нормы)
2. **Закон РФ** — алкоголь/табак 18+, кредиты/договоры 18+, работа с 16 (лёгкая — с 14)
3. **Физическое развитие** — самостоятельное плавание с 8, сложный спорт с 13
4. **Когнитивное развитие** — чтение с 8, медитация с 8, абстрактное мышление с 13
5. **Социальные нормы** — свидания с 13, клубы 18+, бар 18+

### Архитектурные правила для цельной модели

1. **`ageGroup` должен стать явным для всех player-facing действий**. Не полагаться на `undefined`, кроме временного переходного периода.
2. **`undefined` не должен быть бизнес-смыслом**. Для «доступно всем» лучше использовать явный `AgeGroup.INFANT`, чтобы новые действия не обходили фильтрацию случайно.
3. **`ageGroup` = минимальный возраст самостоятельного выбора действия**, а не просто возможность присутствовать рядом с родителем. Иначе модель начинает противоречить сама себе.
4. **Юридический порог и жизненный этап надо разделить явно**. Если остаётся группа `YOUNG = 16–18`, это игровой этап, а не юридическая взрослость.
5. **Нельзя считать `KID` “неиспользуемым” глобально**. Он не должен использоваться в новом маппинге adult-actions, но уже используется в childhood subsystem и должен быть либо сохранён, либо последовательно мигрирован.
6. **Разблокировка вкладки и наличие действий должны проектироваться вместе**. Недопустимо, чтобы вкладка открывалась раньше, чем у неё появляются реалистичные действия.

### Ключевая развилка, которую нужно зафиксировать до реализации

Сейчас в плане одновременно присутствуют два несовместимых тезиса:

- **реализм по закону РФ**: взрослые права и договоры с 18 лет;
- **текущая игровая сетка**: `YOUNG = 16–18`, `ADULT = 19+`.

Это создаёт скрытую логическую дыру: **18-летний персонаж юридически взрослый, но по `AgeGroup` ещё не ADULT**.

Нужно выбрать один из двух вариантов и зафиксировать его в плане до начала массовой разметки действий:

**Вариант A — приоритет реализма законам**
- Изменить диапазоны на `YOUNG = 16–17`, `ADULT = 18+`.
- Это самый логичный вариант, если действия типа аренды, кредита, алкоголя, страхования должны открываться по реалистичному правовому порогу.

**Вариант B — приоритет текущей игровой прогрессии**
- Оставить `YOUNG = 16–18`, `ADULT = 19+`.
- Но тогда ввести отдельный критерий `legalMinAge`, иначе часть «взрослых» действий будет нереалистично недоступна в 18 лет.

**Рекомендация:** выбрать **Вариант A**, потому что он проще, реалистичнее и снимает необходимость плодить вторую систему ограничений.

Часть проблем решается не только сменой `ageGroup`, но и **уточнением самих названий**:

- если действие подразумевает участие с родителями, это лучше отражать в тексте;

### 6. Практическое правило для финальной разметки

При споре между двумя возрастами использовать такой порядок приоритета:

1. **Юридический реализм**
2. **Самостоятельность персонажа**
3. **Понятность названия карточки**
4. **Нужда заполнить страницу контентом**

Если действие нужно только для того, чтобы страница не была пустой, лучше:
- добавить более подходящее по возрасту действие;
- или поднять unlock вкладки;
- но не оставлять нереалистичное назначение.

## Исследование контекста (факты кода)

### Существующая инфраструктура

1. **Тип `AgeGroup`** определён в [`types.ts`](src/domain/balance/actions/types.ts:7). Комментарии в enum содержат ошибки и сейчас вводят в заблуждение. Отдельно важно: `KID` **не должен использоваться в новом маппинге adult-actions**, но **уже используется** в childhood-модулях, поэтому это не «мёртвый enum».

2. **Поле `ageGroup`** уже существует в [`BalanceAction`](src/domain/balance/actions/types.ts:47) как необязательное: `ageGroup?: AgeGroup`.

3. **Фильтрация уже реализована** в [`useAgeRestrictions/index.ts`](src/composables/useAgeRestrictions/index.ts:155):
   - `isActionAvailable()` — проверяет `action.ageGroup <= ageGroup.value`
   - `filterActionsByAge()` — фильтрует массив действий
   - Автоматически применяется в `useActions.getActionsByCategory()`.

4. **UI** поддерживает блокировку вкладок (`GameNav.vue`), но не блокирует отдельные карточки — они просто фильтруются.
5. **План теперь исходит из того, что сон должен жить в time-system**, а не в наборе actions. Значит часть текущих `fun-actions` — это не возрастная задача, а архитектурный долг.

### Критические несоответствия

1. **`TEEN` никогда не возвращается** — `getAgeGroup()` перескакивает с KID (13-15) на YOUNG (16-18). **Все TEEN-действия будут скрыты.**
2. **`child-actions.ts`**: `kid_try_cigarette` (стр. 487) имеет `ageGroup: AgeGroup.TEEN`, но находится в секции KID (8–12) — **ошибка**.
3. **Дублирование `AGE_RULES`**: определены в [`useAgeRestrictions/index.ts`](src/composables/useAgeRestrictions/index.ts:17) и [`age-unlocks.ts`](src/composables/useAgeRestrictions/age-unlocks.ts:13) — нужен рефакторинг.
4. **`child-actions.ts` не интегрирован** с общей фильтрацией `useActions` — используется отдельно в `ChildhoodView.vue`.
5. **`social_family_dinner` и `social_family_holiday`** имеют `requiresRelationship: true` — логическая ошибка (семейный ужин не требует партнёра).
6. **Пустые страницы**: Career для CHILD (8-12), Finance для YOUNG (16-18) — если все действия старшей группы.
7. **`age-unlocks.ts` использует числовые литералы** вместо `AgeGroup` enum — хрупкий код.
8. **`childhood-balance.ts` и `ChainResolverSystem` содержат старые диапазоны возрастов**:
   - `TODDLER = 1–3`, `CHILD = 4–7`, `KID = 8–12`, `ADULT = 18+`
   - Это уже расходится с новым планом (`TODDLER = 4–7`, `CHILD = 8–12`, `ADULT = 19+` в текущей версии документа).
9. **`CHILDHOOD_EVENTS_BY_AGE_GROUP` маппит `PRESCHOOL_EVENTS` в `AgeGroup.CHILD`**, а не в `AgeGroup.TODDLER`. Если оставить это как есть, терминология возрастов останется внутренне противоречивой даже после фикса UI-экшенов.

### Что ещё нужно улучшить в плане, чтобы он был реалистичным и самосогласованным

1. **Сделать один источник истины для возрастных диапазонов**:
   - `AgeGroup`
   - `getAgeGroup()`
   - `AGE_GROUP_RANGES`
   - `TAB_UNLOCK_AGE`
   - childhood events / chain resolver

2. **Явно отделить “сам выполняет” от “может участвовать с родителями”**:
   - Например, кино, театр, музей, пляж, концерт для детей реалистичны.
   - Но если карточка в игре означает именно самостоятельное действие игрока, часть `TODDLER`-назначений может оказаться завышенно щедрой.

3. **Зафиксировать правило для страниц с покупками, финансами и жильём**:
   - `shop` допускает покупки с родителями/на карманные деньги;
   - `finance` и `home` чаще означают юридически значимые решения;
   - Поэтому одинаковый подход “всё по минимальному возрасту участия” здесь даст нереалистичные результаты.

4. **Добавить защиту от будущих регрессий на уровне данных**:
   - тест, что во всех action-файлах у каждого action есть явный `ageGroup`;
   - тест, что `TAB_UNLOCK_AGE` не открывает пустую вкладку;
   - тест, что `ageGroup` не использует запрещённые значения для adult-actions, если решено исключить `KID`.
5. **Отделить “автоматические потребности” от “player-facing actions”**:
   - сон не должен участвовать в age-фильтрации как обычная карточка;
   - age-модель должна покрывать только реальные действия выбора игрока.
