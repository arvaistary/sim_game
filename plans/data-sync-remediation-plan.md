# План устранения рассинхронизации данных (Data Sync Remediation)

## Оценка плана

**Сильные стороны**

- Чётко зафиксирован **канонический контракт** (работа, карьера, время, статы) и блокирующая **фаза 0** до принятия правил в коде и в плане.
- Хорошее разбиение на фазы: сначала чтение и снимки (1A–1B), затем мутации времени (2), реактивность логов (3), закрытие legacy (4), типы и тесты (5).
- Есть **DoD по фазам**, обязательный минимум тестов и **ручной чеклист** — это снижает риск регрессий при рефакторинге стора и ECS.
- Раздел **Suggested Rollout** разбивает работу на PR с осмысленными границами.

**Пробелы и риски**

- План сфокусирован на **работе, времени, логах**; не описывает явно **единый путь выполнения балансовых действий** (покупки, отдых и т.д.) — см. [дополнения](#дополнения-к-базовому-плану).
- **Out of Scope** исключает «новую механику», но **исправление рассинхрона действий/обратной связи UI** может трактоваться по-разному — в дополнениях зафиксировано, что уже сделано по UX.
- Зависимость от **ручной QA** в конце: имеет смысл явно назначить ответственного или зафиксировать сценарий в CI/smoke, где возможно.

**Рекомендация:** продолжать закрывать оставшиеся пункты чеклиста реализации; параллельно держать в уме дополнения ниже как часть «контракта игрок ↔ состояние».

---

## Решение

- Результат аудита: **12 уникальных точек рассинхрона** в `src` (больше порога в 5).
- По правилу задачи требуется **план**, а не немедленный широкий рефакторинг.

## Цель

Обеспечить, чтобы весь игровой UI и системы **читали и писали в согласованные источники состояния** (`work`, `career`, `time`, `stats`) и не расходились с legacy/рантаймом.

## Канонический контракт данных (ворота фазы 0)

Перед фазами реализации нужно зафиксировать политику «источника истины»:

- **Работа в рантайме:** компонент `WORK_COMPONENT` — поля `id`, `name`, `salaryPerHour`, `requiredHoursPerWeek`, `workedHoursCurrentWeek`, `pendingSalaryWeek`, `schedule`, `employed`.
- **Карьера в рантайме:** плоские поля в `CAREER_COMPONENT` должны **зеркалировать** `WORK_COMPONENT` для безопасного чтения в UI.
- **Legacy-поле:** `career.currentJob` — **только снимок для совместимости** со старыми сейвами и переходным UI; не основной источник для новой логики.
- **Время в рантайме:** `TIME_COMPONENT`, нормализованный через `TimeSystem.normalizeTimeComponent`.
- **Производные поля стора:** не должны содержать **независимых формул**, конфликтующих с `TimeSystem`.

**Условие ворот:** работа по фазам не начинается, пока контракт не принят в комментариях к коду и в этом плане.

## Ключевые выявленные проблемы

1. `ActivityLogCard` раньше подгружал записи только при монтировании и мог показывать устаревшие данные.
2. В `WorkPeriodSystem` был запасной путь, обновлявший только `totalHours` без полной нормализации времени.
3. В `game.store` были производные поля (`currentJobName`, `gameDays`, `age`) с формулами, не совпадающими с рантайм-системами.
4. Разные UI-компоненты по-разному резолвили данные о работе (`work`, `career.currentJob`, `career`) — риск разного отображения.
5. Диффы результата работы в `WorkButton` могли читать недельные часы из другого источника, чем модалка с данными о работе.
6. Legacy `career.currentJob` не везде обновлялся после смены работы/карьеры.
7. Типы в `domain/engine/types` частично описывали старую структуру и могли вводить в заблуждение.

## План выполнения

### Фаза 0 — Совместимость и контракт миграций (блокирующая)

1. Описать политику совместимости для save/load:
   - в старых сейвах может быть только `currentJob`;
   - рантайм всегда материализует `work` и плоский `career`;
   - `career.currentJob` при необходимости пересобирается из канонических полей.
2. Задокументировать поведение при отсутствии полей (fallback по умолчанию) в `PersistenceSystem` и/или при инициализации фасада.
3. Добавить **одну** функцию-хелпер миграции формы работы (единая точка).

**DoD**

- Одна задокументированная функция приводит любую legacy-форму работы к канонической рантайм-форме.
- Загрузка старого сейва не оставляет мир без `work`, если в `currentJob` был `id`.
- Ни один компонент **не обязан** читать напрямую `career.currentJob` для работы.

### Фаза 1A — Единая модель чтения в UI (высокий приоритет)

1. Ввести небольшой read-helper для снимков текущей работы и времени на уровне application/domain.
2. Перевести дашборд и блоки карьеры на этот helper (вместо цепочек `work ?? legacy ?? career` в каждом компоненте).
3. Убрать или пометить устаревшим публичное использование `store.currentJobName`.

**DoD**

- Профиль, панель карьеры и модалка работы показывают **одинаковые** имя работы, зарплату и прогресс по неделе.
- Ни один компонент в объёме не читает `store.currentJobName`.
- Нет копипасты цепочки fallback в трёх и более компонентах.

### Фаза 1B — Нормализация чтения времени в store/UI

1. Заменить конфликтующие производные поля стора (`gameDays`, `age`) на значения из канонического `TIME_COMPONENT`.
2. Убедиться, что подписи времени в профиле используют только нормализованные значения.
3. Убрать тихие дефолты, скрывающие невалидное время (`|| 1` там, где ноль — реальное состояние), если это не задумано явно.

**DoD**

- «Время профиля» совпадает с полями `TIME_COMPONENT` после каждого действия.
- В сторе не остаётся независимых формул возраста/дня для UI рантайма.

### Фаза 2 — Единообразный поток изменения времени

1. Убрать или заменить обходные пути мутации времени, минуя `TimeSystem.normalizeTimeComponent`.
2. Согласовать производные стора с каноническими полями времени из `TIME_COMPONENT`.
3. Обеспечить, чтобы любое действие, тратящее часы, давало согласованные `weekHoursRemaining`, `dayHoursRemaining` и возраст.

**DoD**

- Любое действие с тратой часов обновляет все подписи времени согласованно за один тик.
- `WorkPeriodSystem` не мутирует только `totalHours` без нормализации.
- Путь `advanceHours` — единственный предназначенный способ продвижения времени для игровых действий.

### Фаза 3 — Логи: реактивность и детерминизм

1. Перевести `ActivityLogCard` с снимка при `onMounted` на реактивный источник (`computed` или watcher по world/логу).
2. Держать один экземпляр `ActivityLogSystem` на мир и проверить отсутствие дублирующих подписчиков.
3. Подтвердить: одно действие пользователя → одна запись активности, если не ожидается иное.

**DoD**

- Новые записи видны сразу без перезагрузки страницы.
- Нет дублей от одного действия.
- Список активности обновляется после работы/восстановления/финансов в той же сессии без навигации.

### Фаза 4 — Закрытие legacy-дрейфа

1. Зафиксировать политику для `career.currentJob`:
   - зеркалировать из канонических полей после каждой смены работы, **или**
   - хранить только для персистентности и запретить чтение в рантайм-UI.
2. Обновить переходы карьеры/работы (`changeCareer`, продвижение, загрузка/миграция) под эту политику.
3. Ужесточить согласованность save/load для дублирующихся полей времени/работы.

**DoD**

- После загрузки старых сейвов нет «разъехавшегося» состояния работы между страницами.
- Имя/зарплата/часы совпадают на дашборде и в карьере после повышения/смены/перезагрузки.
- Раундтрип `save → reload` сохраняет канонические поля работы и времени.

### Фаза 5 — Типы и защита от регрессий

1. Обновить `domain/engine/types` под реальные рантайм-структуры.
2. Добавить целевые тесты:
   - согласованность профиля и карьеры;
   - прогресс времени при работе;
   - актуальность лога активности;
   - совместимость со старыми сейвами.
3. Добавить краткий чеклист в QA-доках для критичных к синхронизации сценариев.

**DoD**

- CI ловит ключевые регрессии синхронизации до мержа.
- Новый код в объёме компилируется без опоры на устаревшие допущения о типах компонентов.

---

## Обязательный минимум тестов (до мержа)

1. **Unit:** действие работы в `WorkPeriodSystem` продвигает нормализованное время и обновляет недельный прогресс.
2. **Unit:** резолвер снимка работы возвращает одинаковый payload для дашборда и карьеры.
3. **Unit или integration:** источник данных `ActivityLogCard` обновляется реактивно после одного события активности.
4. **Integration:** загрузка старого сейва только с `currentJob` → UI показывает согласованную работу в профиле и в панели карьеры.
5. **Smoke (ручной):** одна смена 8 ч с дашборда → перезагрузка сейва → проверка согласованности профиля/карьеры/лога/времени/статов.

## Предлагаемый порядок внедрения (PR)

1. **PR-1A:** Фаза 0 + Фаза 1A (контракт + модель чтения в UI).
2. **PR-1B:** Фаза 1B + Фаза 2 (нормализация чтения/записи времени).
3. **PR-2:** Фаза 3 (реактивные логи + защита от дублей).
4. **PR-3:** Фаза 4 + Фаза 5 (закрытие legacy + типы/тесты).

## Ответственность (предложение)

- **Domain/Engine:** один ответственный за `WorkPeriodSystem`, `TimeSystem`, `CareerProgressSystem`, `PersistenceSystem`.
- **UI/Store:** один ответственный за компоненты дашборда/карьеры и `game.store`.
- **Проверка:** общий чеклист подписывают оба перед мерджем.

## Риски

- Старые сейвы с неполными полями могут вскрыть скрытые допущения при чистке кода.
- Часть UI сейчас опирается на fallback; строгая канонизация раньше покажет «дыры» в данных (долгосрочно полезно, краткосрочно шумно).
- PR с фазами 1B+2 может раздуться, если смешать чтение/запись времени и чистку стора без жёстких границ по файлам.

## Чеклист валидации

- Отработать смену 8 ч с дашборда:
  - одна и та же работа в профиле и на странице карьеры;
  - прогресс недельных часов совпадает везде;
  - остаток часов недели обновляется сразу;
  - полосы статов обновляются сразу;
  - в логе ровно одна подробная запись.
- Перезагрузить сейв и повторить проверку.
- Сменить карьеру и снова выполнить те же проверки.

## Дополнения к базовому плану

*Ниже — пункты, которые не были в исходном англоязычном документе, но напрямую влияют на ощущение «согласованности» состояния и действий.*

### 1. Единый путь балансовых действий (BalanceAction)

- Все действия из таблиц баланса (магазин, дом, развлечения и т.д.) должны выполняться через **`store.executeAction(actionId)`** → **ActionSystem**, а не через `applyRecoveryAction` / RecoverySystem с произвольным объектом карточки.
- Проверка доступности в UI должна опираться на **`store.canExecuteAction`** (или эквивалент с теми же правилами), чтобы кнопки и движок не расходились.

### 2. Обратная связь после действия (UX и доверие к состоянию)

- После успешного действия или покупки пользователю показывается **модальное окно** с результатом (`showGameResultModal` / `GameModalHost`), а не только тост — иначе неочевидно, что состояние изменилось.
- Ошибки по-прежнему уместно показывать через тосты, чтобы не блокировать экран при отказе.

### 3. Финансы и обучение

- Финансовые сценарии из legacy-списков: канонический путь **`applyFinanceAction`**; по завершении — та же модель модального результата, что и для остальных действий.
- Старт программы обучения: показ результата в модалке согласует ожидания с остальным игровым UI.

### 4. Контроль типов в UI (технический долг)

- При необходимости выровнять типы пропсов списков действий (`ActionCardList` / `BalanceAction` vs `ActionItem`), чтобы `nuxt typecheck` не накапливал расхождения — вспомогательно к фазе 5.
