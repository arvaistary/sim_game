# Рабочие ограничения проекта

## Concurrency limit: 2 одновременных запроса

Текущая модель/профиль поддерживает максимум **2 одновременных запроса** (Concurrency limit = 2).

### Правила для Orchestrator

1. **Не запускать более 2 подзадач одновременно** — если нужно выполнить 3+ независимых задачи, разбивать на батчи по 2.
2. **Ждать завершения перед запуском новых** — после `new_task` дожидаться результата, прежде чем создавать следующую.
3. **Последовательные зависимости** — если задача B зависит от результата задачи A, запускать строго последовательно (A → результат → B).
4. **Параллельные независимые задачи** — если задачи не зависят друг от друга, можно запускать до 2 одновременно.
5. **Планировать с учётом лимита** — при разбиении большой задачи на подзадачи, группировать их так, чтобы минимизировать количество «раундов» (rounds) общения.

### Рекомендуемый паттерн

```
Раунд 1: [Задача 1] + [Задача 2]  (параллельно, если независимы)
Раунд 2: [Задача 3] + [Задача 4]  (после завершения 1 и 2)
...
```

Или последовательно, если есть зависимости:
```
Раунд 1: [Задача 1]
Раунд 2: [Задача 2]  (использует результат Задачи 1)
Раунд 3: [Задача 3]  (использует результат Задачи 2)
```

### Для больших задач

При планировании больших задач (более 4-5 подзадач) — объединять мелкие шаги в более крупные подзадачи, чтобы уменьшить количество раундов. Например, вместо «прочитать файл A», «прочитать файл B», «прочитать файл C» — создать одну подзадачу «прочитать файлы A, B, C и составить отчёт».
